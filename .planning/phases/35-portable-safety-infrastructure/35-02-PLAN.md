**1. Create `netsec-skills/scripts/netsec-scope.sh`:**

Create a standalone bash script for scope management with these exact operations:

```bash
#!/usr/bin/env bash
# netsec-scope.sh -- Portable scope management for pentesting engagements
# Usage: netsec-scope.sh <init|add|remove|show|clear> [target]
#
# Manages .pentest/scope.json in the user's project directory.
# Works from any directory -- resolves project via CLAUDE_PROJECT_DIR > git root > CWD.
```

- `set -euo pipefail`
- `resolve_project_dir()` function (same pattern as hooks: CLAUDE_PROJECT_DIR > git > pwd)
- `PROJECT_DIR="$(resolve_project_dir)"` and `SCOPE_FILE="$PROJECT_DIR/.pentest/scope.json"`
- Require `jq` with helpful error: `command -v jq >/dev/null 2>&1 || { echo "Error: jq is required. Install: brew install jq (macOS) or apt install jq (Linux)"; exit 1; }`

Case statement for operations:
- `init`: `mkdir -p "$(dirname "$SCOPE_FILE")"`, write `{"targets":["localhost","127.0.0.1"]}`, print "Scope initialized at $SCOPE_FILE", `jq . "$SCOPE_FILE"`
- `add`: Require `$2`, check scope file exists (suggest init if not), use `jq --arg t "$2" '.targets += [$t] | .targets |= unique'` with tmp file swap, print "Added '$2' to scope", show targets
- `remove`: Require `$2`, check scope file exists, use `jq --arg t "$2" '.targets -= [$t]'` with tmp file swap, print "Removed '$2' from scope", show targets
- `show` (default when no arg): If file exists, `jq . "$SCOPE_FILE"`. If not, "No scope file found. Run: netsec-scope.sh init" and exit 1
- `clear`: Check file exists, write `{"targets":[]}`, print warning that all pentesting commands will be blocked
- `*`: Print usage and exit 1

Make the file executable: `chmod +x`

**1. Rewrite `netsec-skills/hooks/netsec-health.sh` for dual-context:**

The current health check hardcodes paths to `.claude/hooks/` and `.claude/settings.json`. In plugin context, hooks are at `${CLAUDE_PLUGIN_ROOT}/hooks/` and registration is in `hooks.json`. Rewrite to detect and adapt.

Key changes:

**a. Add resolve_project_dir() function** (same as hooks):
```bash
resolve_project_dir() {
  if [[ -n "${CLAUDE_PROJECT_DIR:-}" ]]; then
    echo "$CLAUDE_PROJECT_DIR"
  elif git rev-parse --show-toplevel 2>/dev/null; then
    :
  else
    pwd
  fi
}
PROJECT_DIR="$(resolve_project_dir)"
```

**b. Add context detection:**
```bash
if [[ -n "${CLAUDE_PLUGIN_ROOT:-}" ]]; then
  CONTEXT="plugin"
  HOOK_DIR="$CLAUDE_PLUGIN_ROOT/hooks"
  HOOK_CONFIG="$CLAUDE_PLUGIN_ROOT/hooks/hooks.json"
else
  CONTEXT="in-repo"
  HOOK_DIR="$PROJECT_DIR/.claude/hooks"
  HOOK_CONFIG="$PROJECT_DIR/.claude/settings.json"
fi
```

Print context at the top: `echo "Context: $CONTEXT"`

**c. Adapt Hook Files section:**
- Check `$HOOK_DIR/netsec-pretool.sh` and `$HOOK_DIR/netsec-posttool.sh`
- Labels should say "(plugin: hooks/)" or "(.claude/hooks/)" based on context

**d. Adapt Hook Registration section:**
- Plugin: check `jq -e '.hooks.PreToolUse' "$HOOK_CONFIG"` (hooks.json)
- In-repo: check `jq -e '.hooks.PreToolUse' "$HOOK_CONFIG"` (settings.json)
- Label: "registered in hooks.json" or "registered in settings.json"

**e. Keep Scope Configuration identical** (always checks `$PROJECT_DIR/.pentest/scope.json`)

**f. Keep Audit Infrastructure identical** (always checks `$PROJECT_DIR/.pentest/`)

**g. Update Dependencies section:**

- REMOVE the "bash version >= 4.0 (associative arrays)" check -- the portable hooks no longer need bash 4.0. Replace with a simpler check: "bash version detected" that reports the version but doesn't fail:

check "jq is installed" \

# Informational: report bash version (no longer a hard requirement)

Replace `((PASS++))` with `PASS=$((PASS + 1))` and `((FAIL++))` with `FAIL=$((FAIL + 1))` in the `check()` function to avoid set -e exit on zero-value increment. Also replace `CHECKS+=()` array append with a CHECKS_STR string accumulator approach that works on bash 3.2 (or keep the array since arrays work on 3.2, just not associative arrays -- regular indexed arrays are fine).

**i. Adapt guided repair for plugin context:**
- In plugin context, don't offer to fix hook file permissions (they're managed by the plugin system)
- In in-repo context, keep existing repair suggestions
